Conheça as novidades do HTML 5.2

Em 14 de dezembro de 2017 o HTML 5.2 tornou-se uma recomendação oficial W3C (REC). Quando uma especificação entra no estágio REC, isto significa que ela já recebeu o endosso dos membros e do diretor do W3C, e assim o W3C oficialmente recomenda sua utilização por usuários, e sua implementação por autores de sites.

A partir do estágio REC, qualquer coisa nova deve ter no mínimo 2 implementações independentes. Isto torna o momento perfeito para nós, web developers, iniciarmos a implementação de qualquer nova funcionalidade.

No HTML 5.2, existe um certo número de adições e remoções, as quais podem serem vistas na página de alterações. Neste artigo irei abordar algumas das alterações que julgo serem as mais relevantes para minhas tarefas de desenvolvimento.

Novas funcionalidades

Um elemento <dialog> nativo

De todas as mudanças no HTML 5.2, estou mais entusiasmado com a introdução do elemento <dialog>, uma caixa de diálogo modal nativa. As caixa de diálogo são bastante comuns na web, mas cada implementação é diferente de alguma forma. As caixas também são realmente difíceis de fazer de uma forma acessível, resultando que a maioria delas são praticamente inacessíveis para usuários que não navegam visualmente na web.

O novo elemento <dialog> pretende mudar isso, fornecendo uma maneira simples de incluir uma caixa de diálogo modal sem ter que se preocupar com muitas das armadilhas. Aqui estão os conceitos básicos.

O diálogo é criado usando um elemento <dialog>:

<dialog>
<h2>Titulo da caixa</h2>
<p>Corpo da mensagem.</p>
</dialog>

Por padrão, a caixa de diálogo está oculta da exibição (e do acesso DOM), a menos que o atributo aberto seja aplicado.

<dialog open>

O atributo open pode ser alternado ao chamar os métodos show() e close(), que estão disponíveis para qualquer HTMLDialogElement.

<button id="open">Abrir caixa</button> <button id="close">Fechar caixa</button> <dialog id="dialog"> <h2>Titulo da caixa</h2>
<p>Corpo da mensagem.</p> </dialog> <script> const dialog = document.getElementById("dialog"); document.getElementById("open").addEventListener("click", () => { dialog.show(); }); document.getElementById("close").addEventListener("click", () => { dialog.close(); }); </script>

De acordo com o caniuse.com elemento <dialog> já possui suporte no Chrome e pode ser habilitado através de uma flag no Firefox.

Usando a API de solicitação de pagamento a partir de iframes

A API de Solicitação de Pagamento é uma alternativa nativa aos formulários de pagamento. Ele pretende fornecer um método padronizado e consistente de fazer pagamentos na web para usuários, movendo o tratamento de recuperação de informações de pagamento para o navegador, em vez de formas de pagamento individuais em cada site.

Antes do HTML 5.2, esses pedidos de pagamento não podiam ser feitos a partir de iframes incorporados em um documento. Isso tornou essencialmente impossível para que soluções de pagamento embutidas de terceiros (exemplo: Stripe, Paystack) utilizassem esta API, uma vez que sua interface de pagamento normalmente é tratada dentro de um iframe.

O HTML 5.2 introduziu o atributo allowpaymentrequest que, quando aplicado a um iframe, permitirá que ele use a API de solicitação de pagamento enquanto o usuário estiver na mesma página hospedada.

<iframe allowpaymentrequest>

Tamanho dos ícones da Apple

Para definir ícones de páginas da web, usamos o elemento <link rel = "icon"> no header do nosso documento. Para definir diferentes tamanhos de ícones, usamos o atributo sizes.

<link rel="icon" sizes="16x16" href="path/to/icon16.png"> <link rel="icon" sizes="32x32" href="path/to/icon32.png">

Este atributo, embora meramente consultivo, permite aos navegadores decidirem qual o tamanho de ícone que eles devem usar se vários tamanhos estiverem disponíveis, especialmente porque a maioria dos navegadores possuem seu próprio tamanho de ícone padrão.

Antes do HTML 5.2, o atributo sizes só era válido se a rel do link fosse icon. No entanto, os dispositivos iOS da Apple não suportam o atributo sizes. Para contornar isso, a Apple apresentou um rel específico do dispositivo, o ícone do apple-touch-icon, que poderia ser usado para definir o ícone usado em seus dispositivos.

No HTML 5.2, a especificação agora permite que o atributo sizes seja usado se o relacionamento for apple-touch-icon, não mais um icon. Isso nos permitirá fornecer diferentes tamanhos de ícones para diferentes dispositivos Apple. Embora, tanto quanto eu sei, os dispositivos Apple ainda não suportam o atributo sizes, essa mudança será útil para o futuro quando eles eventualmente o fizerem.

Novas práticas agora são válidas

Além de novos recursos, o HTML 5.2 validou algumas práticas na escrita de HTML que anteriormente eram inválidas.

Múltiplos elementos <main>

O elemento <main> representa o conteúdo principal de uma página da web. Enquanto o conteúdo repetido em várias páginas pode ser colocado em headers, sections ou qualquer outro elemento, o elemento <main> é reservado para o conteúdo específico e exclusivo da página específica. Consequentemente, antes do HTML 5.2, o elemento <main> precisava ser exclusivo dentro do DOM para que a página fosse válida.

Com a prevalência das SPAs (Single Page Aplications), no entanto, manter esta regra pode ser difícil. Pode haver casos em que existem vários elementos <main> no DOM, mas apenas um é mostrado ao usuário em qualquer momento.

Com o HTML 5.2, agora podemos ter vários elementos <main> em nosso código HTML, desde que apenas um seja visível para o usuário em qualquer momento. Todos os elementos extras devem ser ocultos usando o atributo hidden.

<main>...</main> <main hidden>...</main> <main hidden>...</main>

Como sabemos, existem várias maneiras de ocultar um elemento com CSS. No entanto, quaisquer elementos adicionais <main> devem ser ocultos usando o atributo hidden. Qualquer outro método para ocultar o elemento (exemplo: display: none; ou visibility: hidden;) não será válido.

Style no <body>

Normalmente, o CSS inline definido usando o elemento <style> é colocado dentro do <head> do documento HTML. Com o aumento no desenvolvimento de componentes, os desenvolvedores viram o benefício de se escrever colocando styles junto com os elementos html para quais são relevantes.

No HTML 5.2, a definição de um bloco <style> inline, em qualquer lugar dentro do <body> de um documento HTML, agora é válida. Isso significa que podemos ter estilos mais próximos de onde eles são usados.

<body> <p>Paragrafo 1</p> <style> p { color: red; } </style> <p>Paragrafo 2</p> </body>

No entanto, ainda é observado que os styles deveriam, de preferência, ser colocados no <head>, por motivos de desempenho. De acordo com a especificação:

Um elemento de estilo deve preferencialmente ser usado no header do documento. O uso de estilo no body do documento pode causar restyling, trigger layout e / ou causar repainting e, portanto, deve ser usado com cuidado.

Também deve ser observado que, como mostrado no exemplo, os estilos não respeitam o escopo. Um estilo inline definido mais tarde no documento HTML ainda se aplicará aos elementos definidos antes dele, razão pela qual ele pode desencadear o repainting.

Cabeçalhos em uma <legenda>

Nos formulários, o elemento <legend> representa uma legenda para os campos de formulário dentro de um <fieldset>. Antes do HTML 5.2, o conteúdo de uma legenda tinha que ser plain text. Agora, podemos incluir elementos de cabeçalho.

<fieldset> <legend><h2>elemento h2</h2></legend> <!-- Form fields for basic information --> </fieldset>

Isso é realmente útil para quando queremos usar o elemento fieldset para agrupar diferentes seções de um formulário. Em casos como este, faz todo o sentido usar cabeçalhos, o que tornaria essas seções de formulário mais facilmente navegáveis para usuários que dependem de um contorno de documento.

Recursos removidos

No HTML 5.2, alguns elementos foram removidos, a saber:
  • keygen: usado para ajudar a gerar chaves públicas para formulários
  • menu e menuitem: usado para criar menus de navegação ou contexto

Nenhum children inline, flutuante ou em bloco de <p>

No HTML 5.2, os únicos children válidos de um elemento <p> devem ser o conteúdo de frases. Isso significa que os seguintes tipos de elementos não devem mais ser aninhados dentro de um parágrafo:

  • Blocos inline
  • Tabelas inline
  • Blocos posicionados com float e position

Sem Documentos Esquisitos

Finalmente, podemos dizer adeus aos seguintes tipos de documentos strict:

<!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.01//EN" "http://www.w3.org/TR/html4/strict.dtd">

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Strict//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-strict.dtd">
